- Navette delay impact: Recognized that internal shuttle 72km wait time could exceed transport savings

#### What was NOT actively falsified?

❌ **Gaps in Falsification:**
1. **No contrary supplier data tested:** Agent analyzed only 8 suppliers in favorable geography. What if suppliers concentrated around D2 only? Not tested.
2. **No timing variation analysis:** Hypothesis assumes consistent transport times. What if D2 has consistently faster delivery (informal relationship with closer supplier)? Would proximity still matter? Not tested.
3. **No volume-stacking test:** Hypothesis assumes each pickup independent. What if 5 small orders grouped = 1 large order? Proximity rule might fail. Not explored.
4. **No seasonal falsification:** Winter vs summer supplier availability? Different response time? Not tested.
5. **No failure scenario escalation:** "What would prove proximity WRONG?" → Only vague threshold given (20 km break-even). No testing of what distance threshold would actually fail in practice.

**Confidence in Falsification Attempt: 6/10 (MODERATE)**
- Agent identified one major exception (urgency)
- Agent documented edge cases (wood density)
- Agent did NOT actively seek contrary evidence or stress-test assumptions

**Popper Verdict:** Partial falsification. Hypothesis was refined (with urgency exception), but not truly challenged with alternative scenarios.

## PART E: WISDOM SYNTHESIS - BUDDHA'S MIDDLE WAY

### The Problem with Single-Hypothesis Optimization

If Gedimat optimizes ONLY on Hypothesis 1 (proximity):
- ❌ Risk: Consolidation delays accumulate → Customers frustrated by 2-3 day waits
- ❌ Risk: Small urgent orders deprioritized for distance-first logic → Emergency orders delayed
- ❌ Result: Cost savings (12-15%) but satisfaction drops (long waits visible to customers)

If Gedimat optimizes ONLY on Hypothesis 2 (volume consolidation):
- ❌ Risk: 2x/week Milkrun becomes bottleneck → No flexibility for urgent pickups
- ❌ Risk: Supplier cascade failures → One late supplier delays all 3 partners
- ❌ Result: Theoretical 8-12k€ savings but operational chaos when Milkrun fails

If Gedimat optimizes ONLY on Hypothesis 3 (delivery speed):
- ❌ Risk: Every order gets immediate individual transport → Cost explodes (opposite of optimization)
- ❌ Risk: No consolidation = Mediafret rates at highest tier (15-25€/t) instead of 8-10€/t
- ❌ Result: Customer satisfaction high but company margin erased

### The Middle Way: Multi-Criteria Balanced Scoring

**Proposal: Decision Matrix for Depot Selection**

```
Score(Decision) = (Distance_Score × 0.35)
                + (Volume_Score × 0.30)
                + (Urgency_Score × 0.35)

Where:
  Distance_Score = (MIN_distance / Actual_distance) × 100
  Volume_Score = (MIN_volume_threshold / Actual_volume) × 100
  Urgency_Score = (Customer_delay_tolerance - Expected_wait) / Max_delay_tolerance
```

**Interpretation:**
- **Distance weight (35%):** Proximity matters for cost optimization
- **Volume weight (30%):** Consolidation benefits recognized but not paramount
- **Urgency weight (35%):** Customer satisfaction = equal priority to cost
- **Dynamic:** Each factor contributes equally (35%-35%-30%) with flexibility case-by-case

**Decision Rules with Middle Way:**
1. **Standard Case (no urgency):** Optimize distance + volume equally
   - If D2 (Méru) is 20% closer but D1 (Évreux) has 70% volume → Score them both, pick winner

2. **Urgent Case (customer delay intolerant):** Urgency weighted to 50%, distance to 25%, volume to 25%
   - Customer says "need in 24h" → Score shifts to prioritize speed over consolidation

3. **Exception: Pilot Milkrun (Hypothesis 2):** Special scoring during consolidation window
   - If consolidation meets >3 supplier threshold, create dedicated Milkrun slot (separate from urgency scoring)
   - Goal: 2 tracks simultaneously (Milkrun + standard) until Milkrun stabilizes

4. **Feedback Loop (Hypothesis 3):** Monthly review of satisfaction vs distance choice
   - Hypothesis 3 baseline established → Correlate delivery method choice with NPS change
   - If NPS drops after proximity optimization → Adjust urgency weight upward
   - If satisfaction stable → Continue with 35%-30%-35% weights
